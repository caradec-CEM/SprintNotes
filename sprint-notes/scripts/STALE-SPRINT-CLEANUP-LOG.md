# Stale Sprint Cleanup Log

## Overview

Cancelled and "Will Not Implement" (WNI) tickets in JIRA continue to be dragged through every subsequent sprint after their cancellation date. This happens because closing a sprint automatically moves all incomplete tickets to the next sprint, regardless of whether they are still active. Over time this accumulates dozens of stale sprint associations per ticket, cluttering sprint boards, inflating sprint scope counts, and making velocity reporting inaccurate.

## API Limitation

**JIRA Cloud does not allow removing closed sprint associations via the REST API.** The sprint field (`customfield_10020`) only accepts active or future sprint IDs when using `set`. Closed sprint entries are treated as immutable historical records. All standard API approaches (PUT fields, update operations, backlog endpoint, agile sprint endpoint) were tested and failed on CP-1472.

**Workarounds:**
1. JIRA Admin UI — remove sprint associations through board/sprint administration
2. ScriptRunner plugin — server-side scripting can bypass API validation
3. Forge/Connect app with admin scope — can use `overrideScreenSecurity`
